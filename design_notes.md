Q1 — Why did you structure your solution the way you did?
I split the pipeline into four components, each with a single responsibility. YouTubeAPIClient handles only HTTP communication with the YouTube API — it knows nothing about what to do with the data, it just fetches and returns it. DataFetcher orchestrates the ingestion process — it calls the client, collects videos and comments across multiple channels, and saves the raw JSON to disk before any processing happens. DataTransformer takes that raw JSON and converts it into flat, structured records — it has no knowledge of where the data came from or where it's going. DataLoader handles only the database interaction — creating tables and inserting records. This separation makes each component easy to test, debug, and replace independently.

Q2 — What would break at scale?
Three things would become problems at 50,000 videos. First, the YouTube API quota — at 10,000 units per day, fetching 50,000 videos with comments would require multiple days or multiple API keys. I'd address this by implementing a quota tracker and spreading requests across days or accounts. Second, the current pipeline loads everything into memory before writing to the database — at 50,000 videos this would cause memory issues. I'd fix this by processing and loading data in batches instead of all at once. Third, the requests are sequential — one video at a time. At scale I'd use concurrent requests with a library like asyncio or concurrent.futures to fetch multiple videos in parallel, while respecting the API rate limits.

Q3 — What would you improve with more time?
The comments fetching is the weakest part of the current solution. Right now I'm only fetching the first page of comments (20 comments per video) with no pagination. With more time I'd implement full comment pagination to collect all comments for each video, and add a retry mechanism with exponential backoff for failed requests instead of just returning None and moving on.
